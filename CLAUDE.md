# sokoe-web — Claude 運用ノート

このファイルは Claude が本リポジトリで作業するときの運用ルール集です。
人間のオーナー（株式会社 sokoe 代表）と Claude が共同で更新していきます。

---

## プロジェクトの基本

- **目的**: 株式会社 sokoe のコーポレートサイト + プロダクト LP（sokoe Day / sokoe AI Lab）
- **技術スタック**: Next.js 16 (App Router) + React 19 + TypeScript strict + Tailwind v4 + pnpm 8.15
- **コード品質**: Biome 1.5.3（lineEnding: lf / single quotes JS / double quotes JSX）
- **デプロイ**: Vercel（main ブランチが自動デプロイ）
- **リポジトリ**: https://github.com/ktsuchida-cell/sokoe-hp.git

### 主な対象読者（ターゲット）

- **施設長・管理者**: 中小規模デイサービス（1 日平均 20〜60 名）の運営責任者
- **副施設長 / ケアマネ**: 現場の意思決定者
- **オーナー会社**: 介護施設を運営する会社の経営層

### 提供価値の核（誤解しないように）

- sokoe Day は **介護記録ソフト（カイポケ等）の代替ではない**
- 既存ソフトはそのまま使い、その隣で sokoe Day が **現場業務を支援する**
- 加算判定・国保連請求は sokoe Day 領域外

## 作業フローの基本

1. **オーナーの依頼を受け取ったら、不明点は質問してから着手する**（推測で進めない）
2. **複数ステップになる場合は TodoWrite で見える化**
3. **type-check / lint / screenshot で検証してから commit**
4. **scripts/ ディレクトリ配下のスクリプトは .gitignore 配下なのでコミットしない**
5. **コミットメッセージは英語、命令形、本文で why を語る**

## デザイン作業の進め方（重要）

> 「文字情報が小さすぎたり、コンバージョンポイントが分かりにくかったり」を避けるため、
> **デザイン改修は必ず "なぜそのサイズ・配置か" の根拠とセットで実装する。**

### 1. 設計ノートの保存先

```
design-notes/
  <section>.md          # 各セクションの設計判断を蓄積
  lp-overall.md         # LP 全体の戦略・CV ファネル・視覚階層
```

各 md ファイルには最低でも以下を記録する：

- そのセクションの **目的**（何を伝えたいか）
- **想定読者**（このセクションを読むのは誰か）
- **問題点とその診断**（なぜ改修したか）
- **改善案の詳細**（具体 Tailwind class までは不要、原則レベル）
- **サイズ・配置の根拠**（視覚階層・読者の認知負荷の観点）

### 2. エージェントチームの運用ルール

複数視点・大きな改修・他社事例の調査が必要なときは、サブエージェントを並列で活用する。

- **デザイン視点**: `subagent_type=general-purpose, model=opus`。視覚階層・余白・色使い・UI パターンを評価
- **UX / 離脱リスク視点**: `subagent_type=general-purpose, model=opus`。CV ファネル・認知負荷・CTA 配置を評価
- **業界 / 他社リサーチ**: `subagent_type=general-purpose, model=opus`。WebFetch を使った競合調査
- **Codex 視点**: `codex exec "..."` を Bash 経由で呼ぶ。Claude と異なる視点を取り入れる

エージェントへのプロンプトでは：

- 対象ファイルのフルパスを渡す
- 現状の問題点（オーナーフィードバック）を引用する
- 期待アウトプット形式（語数・Markdown 構造）を明示する
- 実装はさせない、診断 / 設計案のみ依頼する

エージェントの結果は **必ず design-notes に保存** し、その上で Claude が実装する。

### 3. CV ファネルの基本設計

- **中間 CTA**: 資料ダウンロード（低温向け）
- **最終 CTA**: 無料相談予約 30 分（高温向け）
- **Pricing 推奨プラン**: 無料相談予約

> 「閲覧者が何を求めているか」を常に意識し、文字情報・配置・色・サイズはその目的に従う。

> 登録利用者ベースではないので注意。

| プラン | 価格 | 単位 |
|---|---|---|
| フロア機能（BASE） | ¥700 | 1 日平均利用者 1 名／月 |
| 送迎機能（OPTION） | +¥300 | 1 日平均利用者 1 名／月 |
| 管理者業務（OPTION） | +¥300 | 1 日平均利用者 1 名／月 |

- 初期費用ゼロ、30 日間無料トライアル
- 1 日平均 30 名でフル機能利用時の目安：¥39,000／月

## やってはいけないこと

- 推測で「加算判定」「請求機能」を sokoe Day の機能として書く（領域外）
- 「唯一」「妥協なく」「完全対応」「ゼロ」のような過剰断定（誘導感が出る、オーナー指摘あり）
- LP に重複メトリクスを置く（Solution / SelfUse / Features で同じ数字を再掲しない）
- 顔写真プレースホルダーのまま本番デプロイ（信頼を吹き飛ばす）
- `.gitignore` 配下のファイル（screenshots, scripts）をコミット
