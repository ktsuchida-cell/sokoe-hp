# sokoe Day LP 全体設計ノート

このファイルは `/day-service/` LP 全体の戦略・CV ファネル・視覚階層に関する設計判断を記録するためのもの。
オーナーフィードバックを起点に、四軸（目的 × 想定読者 × CV ファネル × 視覚階層）で全 10 セクション + Sticky CTA を診断する。

> オーナー直近の課題提起（2026-05-13）：
> 「文字情報が小さすぎたり、コンバージョンポイントが分かりにくかったりするかなと思う。
> 全体設計として、どういう目的で、どういう情報を見てる閲覧者が欲しいかをしっかり考えながら
> 作っていかないと…」

参考: `design-notes/positioning.md`（個別セクションの書式例）

---

## 0. LP 全体の前提

| 項目 | 内容 |
|---|---|
| URL | `/day-service/` |
| プロダクト | sokoe Day（デイサービス向け業務支援アプリ） |
| 主要ターゲット | 中小規模デイサービス（1 日平均 20–60 名）の **施設長・管理者**、副施設長 / ケアマネ、運営会社の経営層 |
| プロダクト立ち位置 | カイポケ等の介護記録ソフトの **代替ではなく加算**（請求は既存ソフト、現場業務は sokoe Day） |
| 中間 CTA | 資料ダウンロード（低温向け、Sticky / Hero / Last CTA / Pricing） |
| 最終 CTA | 無料相談 30 分（高温向け、Last CTA / Pricing 推奨） |
| 構成 | Header → Hero → Pain → Solution → Positioning → SelfUse → Features → Pricing → FAQ → AboutFounder → LastCTA → Footer ＋ Sticky |

## 1. 軸 1：セクション別の目的明確化

| # | セクション | 目的（カテゴリ） | 想定読者の状態 | 動かしたい方向 |
|---|---|---|---|---|
| 1 | Header | ナビ / 信頼担保 | 「これは何のサイトか」確認中 | スクロール開始 |
| 2 | DayHero | プロダクト認知 + 第一印象 | 「デイサービス用アプリらしい、で、何が違う？」 | 5 秒で価値判断 → 下にスクロール or 資料 DL |
| 3 | DayPain | 共感獲得 | 「現場のしんどさが言語化されているか確認したい」 | 「あ、自分の施設のことだ」と当事者化 → 解決策を求めて読み進める |
| 4 | DaySolution | 解決策の提示（機能訴求の入口） | 「で、どう解決するのか」 | 「4 つの方向性、なるほど」と納得 → 詳細を見たくなる |
| 5 | DayPositioning | 誤解防止 / 信頼形成 | 「カイポケと競合？乗り換える必要ある？」 | 「併用 OK なら検討の心理的ハードルが下がる」 → 安心して次へ |
| 6 | DaySelfUse | 信頼形成（E-E-A-T） | 「机上の空論じゃない？」 | 「自社施設で本番稼働してるなら本物だ」と確信 → 機能の中身を見たくなる |
| 7 | DayFeatures | 機能訴求（実体提示） | 「具体的に何ができる？」 | 16 機能で網羅性を実感 → 料金が気になる |
| 8 | DayPricing | 検討材料の提示 + CV | 「いくら？うちの規模だと？」 | 概算 ¥39,000 で違和感がなければ「相談予約」または「資料 DL」 |
| 9 | DayFAQ | 障壁解消 | 「契約期間は？データ移行は？」など個別不安 | 残った不安を消し、CV へ |
| 10 | DayAboutFounder | 信頼形成（人格担保） | 「誰がやってる？営業会社じゃないか確認したい」 | 「現役の施設長代理がつくっている」と理解 → 最終 CTA に進む |
| 11 | DayLastCTA | 最終 CV（高温向け） | 「もう判断は付いた、相談してみてもいい」 | 30 分無料相談を予約 |
| (S) | StickyDayCTA | 全文中の低温 CV 受け皿 | 読み続けているが踏ん切り中 | 資料 DL で連絡先を残してもらう |

### 評価まとめ

セクション間の **物語ライン**（共感 → 誤解防止 → 信頼 → 機能 → 価格 → 個別不安解消 → 人格 → CV）は理論的に正しい順序で組まれている。
ただし以下 2 点で「目的のブレ」がある：

- **DaySolution と DayFeatures の境界が曖昧**：Solution の「ケアプランも議事録も AI で」と Features の AI バッジ群が重複説明的。Solution は「方向性」、Features は「機能列挙」と階層を分けているが、読者からは「同じ話の繰り返し」に見えるリスクがある。
- **DayPositioning の重要度が現在の視覚的扱いと釣り合わない**：オーナーが最も誤解されたくないメッセージ（＝請求は替えない）にも関わらず、セクション内部の文字サイズは控えめ。`positioning.md` の改修で結論文は serif で昇格されたが、**他セクションとの相対比較**で見ると依然として静かすぎる可能性がある。

## 2. 軸 2：CV ファネル分析

### 2.1 温度推移マップ

| 地点 | 想定温度 | 主要 CTA の妥当性 |
|---|---|---|
| Hero | 25–40°C（低温） | 「資料 DL」「相談する →」並列。**Primary = 資料 DL** で正解（低温に push しすぎない） |
| Pain | 35–50°C（共感で温度上昇） | セクション内 CTA なし。**問題なし**（共感の流れを途切れさせない） |
| Solution | 45–55°C | CTA なし。**問題なし** |
| Positioning | 50–60°C | CTA なし。**問題なし**（誤解解消で安心感が増す段階） |
| SelfUse | 55–65°C | 末尾「すべての実証事例を見る →」のみ。**CV CTA がないが、ここはまだ早い** |
| Features | 60–70°C | カテゴリごとに「全機能を見る」、末尾「16 機能の詳細を一覧」。**回遊 CTA のみで CV ボタンなし**。ここが弱点候補（後述） |
| Pricing | 70–85°C | 「無料相談で見積もりを聞く」「料金資料をダウンロード」。**バランス良好** |
| FAQ | 65–75°C | 末尾「お問い合わせはこちら →」のみ。**もっと強くてもよい** |
| AboutFounder | 70–80°C | 「代表メッセージ全文を読む」のみ。**CV CTA を置く位置として弱い** |
| LastCTA | 80–95°C | 「無料相談予約」「資料 DL」並列。**Primary = 無料相談で正解** |
| Sticky | 全域 35–80°C | 資料 DL 単独。**戦略 C と整合** |

### 2.2 中間 CTA と最終 CTA のバランス

- **戦略 C（コンテントドリブン型）= 中間は資料 DL、最終は相談予約**、という方針は CLAUDE.md に明記。これに対し現状の配置は **概ね整合**している。
- ただし **Features 終了から Pricing までの間に CTA がない**。Features は LP 内で最も縦に長い区画（カテゴリ 3 つ × 機能カード × メトリクスカード）であり、ここで離脱されると Pricing 以降の高温 CTA に到達しない。Sticky でフォローはできているが、本文側にも軽量な「資料 DL」リンクを置く余地あり。
- **AboutFounder → LastCTA** の流れは強い（人格 → 行動）。ただし AboutFounder 自体には CV ボタンがないので、ここで「代表に直接相談したい」温度が立ち上がっても、次に進む手段が「下にスクロール」しかない。AboutFounder の末尾にもう一段の控えめな相談 CTA があると、温度を逃さない。

### 2.3 StickyDayCTA の妥当性

| 観点 | 評価 |
|---|---|
| 出現タイミング（`scrollY > 600`） | **適切**。Hero を抜けたあと、Pain で共感が立ち上がる手前から表示開始 |
| 非表示タイミング（Footer まで 1200px） | **適切**。LastCTA と二重表示にならない |
| コピー「機能・料金・導入事例を、まとめてご覧いただけます」 | **やや弱い**。「資料 = まとめ」というだけで、なぜ今 DL するかの動機が薄い。「3 分で要点把握」「現場の声つき」など具体性があると押せる |
| ボタン文「資料をダウンロード（無料）」 | **無難**。摩擦最小なので戦略的に変える必要は薄い |
| dismiss を sessionStorage で保持 | **正解**。LP 再訪時に消えてしまうと中間 CV を取り逃すので、永続化していない |
| サイズ・色 | デスクトップ右下フローティング、モバイル下端 full-width。**標準的で問題なし** |

### 2.4 Pricing 推奨プラン CTA「無料相談を予約」の効き

- フロア機能カードが `border-2 border-product-orange` + `BASE` バッジで強調されており、視線誘導は機能している。
- ただし **推奨プラン直下に CTA がない**。CTA は 3 カード並列の下、概算例のさらに下にまとめて配置されている。**「このプランで申し込む」想起が、推奨カードのスコープ内で完結しない**。
- 概算例ボックス（¥39,000）は妥当だが、ここから CTA まで縦距離が長く、認知が冷める恐れがある。`mt-12` を `mt-8` 程度に詰めるか、概算例の内部に「この想定で相談する」マイクロ CTA を埋めるとよい。

## 3. 軸 3：視覚階層・サイズ問題（オーナー指摘）

オーナーの「文字情報が小さすぎる」指摘に対し、ソースから具体的に **サイズが小さすぎる / 階層が逆転している箇所** を 5 点抽出した。

### 3.1 サイズ問題 5 箇所

| # | 箇所 | 現状サイズ | 適正サイズ案 | 根拠 |
|---|---|---|---|---|
| S1 | **Hero 判断材料 3 点バッジ**（「カイポケ等の介護記録ソフトと併用 OK」など） | `text-[12px] md:text-[13px]` | `text-[13px] md:text-[14px]` + アイコン `w-4 h-4` | 購買判断のキードライバー 3 点が脚注サイズ。**判断材料は本文と同等以上であるべき** |
| S2 | **DayPositioning カード内本文・チェックリスト** | `text-[13px]`〜`text-[14px]` / リスト `text-[13px] md:text-[14px]` | 本文 `text-[15px]` / リスト `text-[14px] md:text-[15px]` | このセクションは「誤解防止」が目的。読まれなければ目的未達。カイポケ等の固有名詞や役割分担は **本文と同等の可読性が必要** |
| S3 | **DayPricing 料金単位**（「1 日平均利用者 1 名／月（税込）」） | `text-xs`（12px） | `text-[13px] md:text-sm`（13–14px） | 従量課金は **誤読が事故**（登録ベースだと思われると激安に見えてしまう）。`xs` は脚注扱いになり、見落とされる |
| S4 | **DayFeatures カードの説明文** | `text-[12px] md:text-[13px]` | `text-[13px] md:text-[14px]` | 1 機能あたりの説明が 1 行で詰まっており、12px だとアイコン群の隣で「ラベル化」して読まれない。**機能の差別化が伝わらない** |
| S5 | **DayAboutFounder の肩書き** | `text-sm` + `mb-6` | `text-[15px]` / 肩書きと本文の間に区切り | 「現役の施設長代理」「自社運営施設」という **E-E-A-T の核**が肩書き行に詰まっている。役職 3 つを 1 行で読まされている |

### 3.2 ヒエラルキー逆転（脚注サイズなのに実は結論文）

| # | 箇所 | 問題 | 改修方針 |
|---|---|---|---|
| H1 | DayPain 末尾「介護施設を運営する会社として、私たち自身もずっと向き合ってきた課題です」 | `text-sm md:text-base text-stone` の **脚注スタイル** で書かれているが、実体は「sokoe の独自性宣言」＝そのセクションの結論。`positioning.md` で指摘されたのと同じ階層逆転 | `text-base md:text-lg` + `text-charcoal`（または ink）、軽い装飾ライン。`positioning.md` の結論文方式を踏襲 |
| H2 | DayPricing「初期費用は無料です。30 日間無料でお試しいただけます」（末尾） | `text-sm text-mid` で脚注化されているが、これは「契約リスクなし」という **CV 阻害要因の解消メッセージ**。脚注では効かない | CTA の **上**に移動し、`text-base text-charcoal`。または CTA ボタンに「30 日無料」のサブテキストとして付帯 |
| H3 | DayHero「自社運営施設で 2026 年 4 月から本番稼働中」 | `text-sm text-charcoal-muted` で脚注扱い。これは E-E-A-T の核 | `text-[15px] text-charcoal` + 軽いバッジ装飾。あるいは判断材料バッジ群と統合 |
| H4 | DayFeatures メトリクス `caption`（「自社運営施設の実証データから試算」） | `text-[12px] md:text-[13px]` の脚注。**52%・月 10 時間・運転席に** といった核心数字の出典であり、信頼性の担保 | `text-[13px] md:text-[14px]`、可能なら凡例風のアイコンを添える |
| H5 | DayPositioning 結論文（既に改修済み） | `text-[20px] md:text-[24px] lg:text-[26px]` serif で **正しく結論として昇格済み**。これを **基準モデル** として他セクションへ波及させるべき | 参考実装として全セクションへ展開 |

### 3.3 視覚階層全体のテーマ

オーナー指摘の根本原因は「**文字 = 大事じゃないものから順に並べる**」癖が一部セクションに残っていること。
階層原則は次の通り：

```
1. H2 セクション見出し（28–48px serif）
2. 結論文 / 答え（20–26px serif）── positioning.md 方式
3. 本文・リード（16–18px）
4. カード内本文（14–15px）
5. 補助情報・脚注（12–13px）
```

現状、**(2) 結論文レイヤーが存在しないセクション**（Pain・Pricing・Hero）と、**(5) のサイズなのに (2) の意味を持つ文**が混在している。これを直すだけで「小さすぎる」印象は大幅に改善される。

## 4. 軸 4：コンバージョン到達阻害要因

### 4.1 CV ボタンまでの認知負荷

- **縦方向の長さ**：Pain（5 項目）→ Solution（4 カード）→ Positioning（2 カード ＋ ＋チップ ＋ 結論文）→ SelfUse（4 ケース）→ Features（3 カテゴリ × メトリクスカード × 機能カード 7+3+6）→ Pricing（3 プラン ＋ 概算例）→ FAQ（10 問）→ AboutFounder → LastCTA。**Features が単独でもスクロール体感 4–5 画面分**。
- **同じ数字の再掲**：「30 日無料」「1 日平均 30 名で ¥39,000」「自社運営施設」が複数セクションで繰り返される。CLAUDE.md でも「重複メトリクスを置かない」と明記されている。**SelfUse / Features メトリクス / Pricing 概算で重複しやすいので注意**。
- **CTA ラベルの揺れ**：
  - Hero：「資料をダウンロード（無料）」「相談する →」
  - Pricing：「無料相談で見積もりを聞く（30 分）」「料金資料をダウンロード」
  - LastCTA：「無料相談を予約（30 分）」「資料をダウンロード」
  - Sticky：「資料をダウンロード（無料）」
  - **「資料 DL」「料金資料 DL」が混在**、「相談する」「無料相談で見積もり」「無料相談を予約」も微差で揺れている。読者が「これは同じボタン？別物？」と一瞬迷う。**4 種類のラベルを 2 種類（資料 DL / 無料相談 30 分）に統一**したい。

### 4.2 情報密度が高すぎるセクション

| セクション | 密度評価 | コメント |
|---|---|---|
| DayHero | 適切 | テキスト 5 / 7 と画像のバランス。判断材料バッジが小さい点だけ要修正 |
| DayPain | やや高 | 5 項目を縦並びで読ませる。3 項目化 + 2 項目は折りたたみ／省略でもよい |
| DaySolution | 適切 | 4 カード `md:grid-cols-2`。タイトル + サブタイトル + 本文の三段構造が読みやすい |
| DayPositioning | 適切 | 改修済み。`positioning.md` の方針通り |
| DaySelfUse | 適切 | 4 ケースで止めているのが良い。「すべて見る」リンクで逃がしている |
| DayFeatures | **過密** | カテゴリ 3 × 機能 16 × メトリクスカード 3 × カテゴリ別 CTA 3 ＋ 末尾 CTA。**Features 単体でランディング 1 ページに相当**する情報量 |
| DayPricing | やや高 | 3 プラン + 概算例（4 行）+ CTA 2 つ + 注記。概算例の縦長が認知を冷ます |
| DayFAQ | 適切（10 問） | アコーディオンで折りたたまれているので密度は実質低 |
| DayAboutFounder | 低 | 余裕あり。**Portrait プレースホルダーが本番だと信頼吹き飛ぶ**（CLAUDE.md 警告事項） |
| DayLastCTA | 適切 | 高温向けの最終 CV としてシンプルで強い |

### 4.3 競合機能比較表は必要か

- **不要寄り**。理由：
  - sokoe Day は **カイポケ等の代替ではなく加算**（CLAUDE.md 明記）。「機能比較表 = どちらが勝つか」の絵を描いてしまうと、Positioning で立てた「+」関係と矛盾。
  - 比較表は読者に「乗り換え検討」モードを誘発する。今の戦略は「乗り換えなくていい、追加するだけ」。
- **必要があれば**、Positioning の右カード（sokoe Day 側）に「これは sokoe Day **だけ** がやる」という一覧を絞った機能を入れる方が、比較感を出さずに差別化を伝えられる。

### 4.4 その他の阻害要因

- **DayAboutFounder の "Portrait" プレースホルダー**：CLAUDE.md で「顔写真プレースホルダーのまま本番デプロイ → 信頼を吹き飛ばす」と警告されている。**最優先で実写差し替え**。
- **Pricing の従量課金単位**：「1 日平均利用者 1 名／月」を `text-xs` で書いている（S3）。料金体系の誤読は CV の致命傷なので、サイズ修正必須。
- **Sticky のコピー**：「機能・料金・導入事例を、まとめてご覧いただけます」は弱い。**「30 ページの導入ガイド（料金シミュレーション付き）」など具体性**を入れる検討。

| P9 | StickyDayCTA のコピーを具体化（資料の中身を匂わせる） | 中 | 高 | **中** |
| P10 | AboutFounder 末尾に控えめな相談 CTA を追加 | 中 | 高 | **中** |
| P11 | Features 末尾と Pricing の間に資料 DL リンクを追加 | 低–中 | 高 | **低** |
| P12 | DayPain を 5 → 3 項目に絞り、残り 2 項目を別表現に | 低 | 中（コピー再設計） | **低** |
| P13 | DaySolution と DayFeatures の重複説明を整理 | 中 | 低（情報設計の再検討） | **後回し（要設計）** |

### 優先度の読み方

- **最優先（P1–P3）**：本番公開時の信頼・転換に直結。1 週間以内に着手したい。
- **高（P4–P6）**：オーナー指摘「文字が小さい」への直接回答。視覚階層の原則をセクションへ波及させる。
- **中（P7–P10）**：CV ファネルの取りこぼし削減。A/B しながら段階的に。
- **低（P11–P12）**：UX 改善。データを見ながら判断。
- **後回し（P13）**：情報設計の再検討が必要。単独セクションではなく LP 全体の物語ライン見直しと一緒に。
